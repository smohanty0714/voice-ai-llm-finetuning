01 — The Big Picture

Where Does KV Cache Sit in an LLM?

Click any component to learn more. The KV Cache lives inside every attention layer — and it grows with every token.

🔥 KV Cache

Inside every attention layer, the model computes Key and Value vectors for each token. These are stored in the KV Cache so they don't need to be recomputed at the next step. With 80 layers and 128K tokens, this cache consumes ~40 GB — half of an H100 GPU.

How it grows

Token 1: Store K₁, V₁ → tiny
Token 100: K₁..K₁₀₀, V₁..V₁₀₀ → growing
Token 128K: K₁..K₁₂₈ₖ, V₁..V₁₂₈ₖ → 40 GB!
× 80 layers = every layer has its own copy!

🏢 Building Analogy

The LLM is an 80-floor office building. Each floor (layer) has a filing cabinet (KV Cache) that stores notes about every visitor (token) who ever entered. The longer the line of visitors, the bigger every cabinet gets — on every floor.

03 — The Problem

The KV-Cache is the Hidden Memory Monster

// KV-Cache Memory Formula
KV_bytes = 2 × n_layers × n_kv_heads × head_dim
           × seq_len × batch_size × bytes_per_elem

// LLaMA-3-70B (GQA: 8 KV heads, 80 layers, d=128)
per_token = 2 × 80 × 8 × 128 × 2 = 320 KB

// At 128K context, batch=1:
total = 320KB × 131,072 = 40 GB 🔥

That's half of an H100's 80GB HBM for a single request. At batch=4, KV-cache (160GB) exceeds the model weights (140GB).

05 — The Naive Approach Fails

Why Simple Offloading Doesn't Work

Live Simulation — Watch the GPU stall
61%
GPU Stall Time

The Core Problem

Each decode step needs KV for all previous tokens across all layers. Naive offloading fetches them on-demand → GPU idles while waiting.

Need: 40 GB of KV data
Have: 64 GB/s bandwidth
Time: 625 ms per step 😱
But: Compute takes ~15 ms

💡 The Key Insight

Attention is ≥95% sparse. We only need 1–10% of tokens. If we can predict which ones...

🍽️ Restaurant Analogy:
Imagine a chef (GPU) needs ingredients from a warehouse (CPU DRAM). Naive approach: chef waits at the door each time. Smart approach: a sous chef predicts what's needed next and pre-stages ingredients on the counter while the chef cooks. The chef never stops working.

09 — Takeaways & Future

Key Takeaways

1
KV-cache is the bottleneck, not model weights
At 128K context, batch=4: KV-cache (160 GB) > model weights (140 GB) for LLaMA-3-70B
2
Attention sparsity makes speculation viable
96%+ of attention mass recovered with just 3-5% of tokens. Power-law distribution is your friend.
3
Speculative prefetching hides transfer latency
Predict → prefetch → compute in parallel. InfiniGen, SpeCache, ScoutAttention achieve 3–8× speedup.
4
Compression × Offloading = multiplicative gains
4-bit quantize + 20% eviction + tiered offloading → 80× reduction in GPU KV memory footprint.
5
The production stack is ready today
vLLM/SGLang + LMCache + KIVI quantization + prefill/decode disaggregation → 3–5× cost reduction.

Open Challenges

Context-intensive tasks
Multi-needle retrieval breaks sparsity assumption
PCIe bandwidth wall
